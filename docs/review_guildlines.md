# 🧠 Review Guidelines

These guidelines support our tiered code review process in the context of high-velocity, AI-assisted development. The goal is to reduce code-to-merge time without compromising safety or clarity.

---

## 🔍 Tier Definitions

### ✅ Tier 0 — Automated Approval
- **Handled by:** CI + GitHub Actions
- **No human review required**
- **Examples:**
  - Linting or Prettier fixes
  - Comment or variable renaming
  - Generated schema, type, or API definitions
  - Automated test snapshot updates

---

### 👀 Tier 1 — Light Human Review
- **Handled by:** One engineer or peer reviewer
- **Goal:** Sanity check the change, verify it aligns with intent
- **Examples:**
  - Addition of AI-generated helper methods
  - Minor feature toggles or config updates
  - Small UI or UX changes with no system impact
  - Updates to tests, docs, or error messages
- **Checklist:**
  - [ ] Clear intent in PR description
  - [ ] Tests pass and cover change
  - [ ] No hidden architectural impact

---

### 🔎 Tier 2 — Deep Review
- **Handled by:** One or more experienced reviewers with context
- **Goal:** Comprehend impact, identify risks, and validate correctness
- **Examples:**
  - Core logic and business rules
  - Authentication, authorization, or security checks
  - Database migrations or schema changes
  - External API changes or backend contracts
- **Checklist:**
  - [ ] Clear AI rationale and scope
  - [ ] New tests or test plan included
  - [ ] Rollback or failure recovery considered
  - [ ] No unintended coupling across modules

---

## ⚠️ Review Anti-Patterns

Avoid merging code that:
- Was generated by AI with unclear intent or edge case handling
- Lacks test coverage for changed logic
- Touches critical paths without a rollback plan
- Introduces architectural complexity without justification

---

## 🕒 Review SLA

| Tier | Review Time Target | Merging Rules |
|------|--------------------|----------------|
| Tier 0 | Immediate on CI pass | Auto-merged |
| Tier 1 | ≤ 12 hours | 1 reviewer approval |
| Tier 2 | ≤ 48 hours | 1–2 reviewers or senior engineer |

---

## 🧠 Philosophy

- Code must be **understandable by others**, even if generated by an agent
- PRs should favor **clarity over cleverness**
- Review effort should **scale with risk**, not with code size
